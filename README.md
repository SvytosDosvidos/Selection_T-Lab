# Отбор в T-Lab

Программа вычисляет PageRank для невзвешенного ориентированного графа, заданного
списком рёбер в CSV. Основная особенность решения — рёбра не хранятся целиком в
оперативной памяти. Они предварительно разбиваются по файлам на SSD, а затем
последовательно читаются на каждой итерации PageRank.

Текущая версия программы:

- принимает один аргумент — путь к входному CSV;
- использует лимит памяти 128 МиБ;
- автоматически выбирает число потоков по числу аппаратных потоков процессора;
- сохраняет промежуточные файлы в каталог `work`;
- записывает результат в `pagerank.csv`.

## Запуск

Программа принимает ровно один аргумент:

```text
graph_analytics edges.csv
```

Linux, запуск из корня проекта:

```bash
./build/graph_analytics tests/data/small.csv
```

Windows при использовании генератора Visual Studio:

```powershell
.\build\Release\graph_analytics.exe tests\data\small.csv
```

После успешного запуска в текущем каталоге появятся:

- `pagerank.csv` — итоговые ранги;
- `work` — бинарное представление графа и временные данные.

## Формат входных данных

Правила чтения:

- Формат: CSV (первые 2 колонки — id пары вершин from и to), 
  описывающих ребро из from в to;
- ID вершины должен помещаться в диапазон `int32` от `-2147483648` до
  `2147483647`;
- Рёбра не имеют веса;
- Содержимое входного файла (edges.csv):
  ```csv
  from,to
  1,2
  2,3
  3,1
  4,1
  ```

## Формат результата

Результат записывается в `pagerank.csv`:

- Формат: CSV;
- Формат вершин — int32;
- Пример выходного файла (leader_rank.csv):

```csv
vertex,rank
1,0.35 
2,0.25 
3,0.25 
4,0.15
```

## Что вычисляет PageRank

PageRank оценивает важность вершины по входящим ссылкам. Ссылка от важной
вершины даёт больший вклад, чем ссылка от вершины с низким рангом. Вклад
источника делится между всеми его исходящими рёбрами.

Для вершины `v` одна итерация имеет вид:

```text
next[v] = (1 - d + d * dangling_mass) / N
          + d * sum(rank[u] / out_degree[u])
```

где:

- сумма берётся по рёбрам `u -> v`;
- `N` — число вершин;
- `d = 0.85` — damping factor;
- `dangling_mass` — сумма рангов вершин без исходящих рёбер.

Ранг висячих вершин равномерно распределяется по всему графу. Благодаря этому
масса ранга не теряется, а сумма значений остаётся около единицы.

## Почему выбран PageRank

PageRank подходит к условию задачи по нескольким причинам:

- это глобальная метрика ориентированного графа;
- одна итерация требует одного последовательного прохода по рёбрам;
- не нужна матрица смежности размером `N * N`;
- рёбра можно хранить на SSD;
- вычисление следующего вектора можно разделить между потоками;
- алгоритм детерминирован и имеет понятный критерий остановки.

Другие рассмотренные метрики:

- **LeaderRank** близок к PageRank, но добавляет служебную вершину, связанную со
  всеми вершинами. Её можно реализовать неявно, однако PageRank проще объяснить,
  проверить и настроить через переменную `d`.
- **Katz centrality** учитывает пути разной длины. Для сходимости нужно подобрать
  коэффициент меньше величины, обратной спектральному радиусу матрицы смежности,
  Но оценка этого радиуса создаёт дополнительную задачу и риск неудачного параметра.
- **Jaccard** и **Dice** измеряют сходство двух множеств соседей. Это парные
  метрики, а не один ранг каждой вершины. Подсчёт для всех пар может дать
  `O(V^2)` результатов и особенно дорог для гипер-узлов.

## Рассмотренные способы хранения графа

Были рассмотрены три варианта.

### 1. Список смежности в оперативной памяти

Это самый простой вариант, но он требует держать все рёбра в RAM и нарушает
главное условие задания на ограничение памяти.

### 2. Сортировка вкладов на каждой итерации

Для каждого ребра можно записывать пару `(to, contribution)`, сортировать такие
пары на диске и суммировать их. Памяти нужно мало, но дорогую сортировку
нужно будет повторять на каждой итерации.

### 3. Разделы по вершине назначения

Этот вариант реализован в проекте. Каждое ребро заранее записывается в раздел,
который определяется по вершине `to`. Один раздел в конкретный момент
обрабатывает только один поток. Поэтому потоки изменяют разные диапазоны массива
`next`, и для каждого прибавления не нужны `mutex` или `atomic<double>`.

## Подготовка дискового графа

Подготовка выполняется двумя проходами по исходному CSV.

### Первый проход: вершины

ID вершин собираются порциями. Размер порции ограничен долей заданного лимита
памяти. Каждая порция сортируется, повторяющиеся ID удаляются, после чего
порция записывается в отдельный бинарный файл. Затем отсортированные файлы
попарно сливаются внешней сортировкой.

Результатом становится `vertices.bin` с уникальными ID по возрастанию. Позиция ID
в этом массиве используется как плотный номер вершины:

```text
-100       -> 0
7          -> 1
1000000000 -> 2
```

### Второй проход: рёбра и степени

CSV читается повторно. Для каждого ребра оба ID находятся в отсортированном
массиве бинарным поиском. После этого программа:

- увеличивает исходящую степень вершины `from`;
- преобразует исходные ID в плотные номера `uint32`;
- записывает ребро в файл, выбранный по номеру вершины `to`.

Рабочий каталог содержит:

```text
work/
  vertices.bin    отсортированные исходные ID
  degrees.bin     исходящие степени
  edges_0.bin     первый раздел рёбер
  edges_1.bin     второй раздел рёбер
  ...
  graph.info      числа вершин, рёбер и разделов
```

Временный каталог `work/runs` после успешного слияния удаляется.

## Многопоточный PageRank

В RAM хранятся только данные, размер которых зависит от числа вершин:

- исходные ID — `int32`;
- исходящие степени — `uint64`;
- текущий вектор рангов — `double`;
- следующий вектор рангов — `double`.

Рёбра остаются в файлах `edges_N.bin`. Потоки получают номера ещё не
обработанных разделов через атомарный счётчик. Внутри раздела рёбра читаются
последовательно.

Число потоков определяется так:

```cpp
max(1U, thread::hardware_concurrency())
```

Нижняя граница нужна потому, что стандарт разрешает
`hardware_concurrency()` вернуть `0`, если число аппаратных потоков неизвестно.
Число разделов равно не более чем `threads * 4`, не более 64 и не более числа
вершин. Дополнительные разделы помогают распределять работу между потоками.

Гипер-узел не загружается в виде большого списка соседей. Его рёбра остаются
обычными бинарными записями и читаются по одному. Гипер-узел с очень большой
входящей степенью всё же может создать дисбаланс, потому что один диапазон
вершин назначения обрабатывается одним потоком.

## Остановка и проверка корректности

Начальное значение каждой вершины равно `1 / N`. После итерации программа
вычисляет L1-разность:

```text
error = sum(abs(next[v] - rank[v]))
```

Вычисление останавливается при `error <= 1e-8`, но выполняется не более 200
итераций. Если сходимость не достигнута, программа возвращает код `2`.

Во время работы печатаются номер итерации, ошибка и сумма рангов. Дополнительно
проверяются границы плотных ID, принадлежность ребра разделу и общее число
прочитанных рёбер. Обработка рёбер внутри каждого раздела всегда
последовательна, поэтому планировщик потоков не меняет порядок сложения для одной
вершины.

При `0 < d < 1` преобразование PageRank является сжимающим. С учётом обработки
висячих вершин и случайного перехода существует единственная неподвижная точка,
к которой сходится степенная итерация.

## Оценка ресурсов

Пусть `V` — число вершин, `E` — число рёбер, `K` — число итераций, `T` — число
потоков.

- сбор ID внешней сортировкой: в худшем случае `O(E log E)` сравнений;
- преобразование рёбер: `O(E log V)` из-за бинарного поиска ID;
- одна итерация: `O(V + E)`;
- все итерации: `O(K(V + E))`;
- постоянное место на диске: `O(V + E)`;
- оперативная память: `O(V)` плюс ограниченная порция внешней сортировки.

Для рангов, степеней и ID требуется примерно `28 * V` байт без учёта накладных
расходов библиотек и файловых буферов. Код проверяет, чтобы эта оценка занимала
не более 80% от 128 МиБ. Лимит является внутренним бюджетом алгоритма, а не
жёстким ограничением памяти процесса со стороны операционной системы.

## Ограничения реализации

- Все рёбра могут быть больше RAM, но четыре массива состояния вершин должны
  помещаться в заданный размер.
- Изолированную вершину нельзя обнаружить во входе, содержащем только рёбра.
  В результат попадают вершины, встреченные хотя бы в одном ребре.
- Повторяющиеся строки считаются отдельными рёбрами.
- Поддерживается простой двухколоночный CSV без кавычек.
- Плотный ID ищется бинарным поиском. Это экономит память по сравнению с большой
  хеш-таблицей, но увеличивает время подготовки до `O(E log V)`.
- Очень большая входящая степень может создать неравномерную нагрузку между
  target-разделами.
- Бинарные промежуточные файлы рассчитаны на повторное чтение той же программой
  на той же x64-платформе.
